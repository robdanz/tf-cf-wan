# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Terraform project managing Cloudflare Magic WAN IPsec tunnels at scale. Creates 2 IPsec tunnels per site (primary + secondary, one to each Cloudflare Anycast IP), creates static routes for each site's LAN subnets, and generates `output/configure-tunnels.sh` to configure the corresponding IPsec tunnels on Aruba EdgeConnect appliances via ECOS REST API.

## Commands

```bash
# Terraform (Cloudflare side)
terraform init                    # Download providers (cloudflare v5, random, local)
terraform validate                # Check syntax
terraform plan
terraform apply -parallelism=1    # REQUIRED: Cloudflare Magic WAN API uses a per-account write lock;
                                  # parallel requests cause 429 "account is being modified elsewhere"
terraform output -raw tunnel_psk                    # Retrieve the shared PSK
terraform output -json tunnel_details               # All tunnel data as JSON
terraform state show 'cloudflare_magic_wan_ipsec_tunnel.tunnels["<site>-pri"]'

# Aruba EdgeConnect ‚Äî site discovery
bash aruba/get_site_details.sh --orchestrator HOST  # Query Orchestrator ‚Üí sites.csv.proposed
bash aruba/demo_orchestrator.sh [--output FILE]     # Simulate Orchestrator for demo (10 NAT'd sites)

# Aruba EdgeConnect ‚Äî configure/remove (Mac/Linux, generated by terraform apply)
bash output/configure-tunnels.sh --username admin --orchestrator ORCH_HOST # required for all sites ‚Äî resolves WAN IP at run time
bash output/remove-tunnels.sh --username admin

# Aruba EdgeConnect ‚Äî configure/remove (Windows PowerShell, generated by terraform apply)
.\output\configure-tunnels.ps1 [-DryRun] [-Sites "site1,site2"]
.\output\remove-tunnels.ps1    [-DryRun] [-Sites "site1,site2"]
```

## Architecture

**Full data flow:**
```
aruba/get_site_details.sh
    ‚îÇ
    ‚îî‚îÄ‚ñ∫ sites.csv.proposed  (review + cp to sites.csv)
            ‚îÇ
            ‚îú‚îÄ‚ñ∫ terraform apply -parallelism=1
            ‚îÇ       ‚îÇ
            ‚îÇ       ‚îú‚îÄ‚ñ∫ Cloudflare: IPsec tunnels (2 per site)
            ‚îÇ       ‚îú‚îÄ‚ñ∫ Cloudflare: static routes (1 per site subnet per tunnel)
            ‚îÇ       ‚îú‚îÄ‚ñ∫ output/cpe-config.csv
            ‚îÇ       ‚îî‚îÄ‚ñ∫ output/configure-tunnels.sh  ‚óÑ‚îÄ‚îÄ auto-generated ECOS script
            ‚îÇ
            ‚îî‚îÄ‚ñ∫ bash output/configure-tunnels.sh --username admin
                    ‚îî‚îÄ‚ñ∫ EdgeConnect appliances (IPsec tunnels configured via ECOS API)
```

**`output/configure-tunnels.sh` is generated by `terraform apply`** from `aruba/configure-tunnels.tftpl` ‚Äî no manual script generation step needed. It has the PSK and tunnel IDs embedded.

### Core transformation in locals.tf

1. `csvdecode()` reads `sites.csv` into a site map keyed by `site_name`
2. Each site is flattened into 2 tunnel entries (`<site>-pri`, `<site>-sec`) via `flatten()`
3. `/31` inside addresses are allocated from the supernet using `cidrsubnet(supernet, 9, site_index * 2 + offset)` ‚Äî the explicit `site_index` column (not row position) controls IP allocation so CSV reordering is safe
4. Within each /31: Cloudflare = `cidrhost(..., 0)` (even/lower), CPE = `cidrhost(..., 1)` (odd/upper)
5. Automatic return routing is enabled on all tunnels (`automatic_return_routing = true`) ‚Äî no static routes needed

### Resource pattern in tunnels.tf

Single `cloudflare_magic_wan_ipsec_tunnel` resource with `for_each = local.tunnel_definitions`. One `random_password` generates a shared PSK (`special = false` for CPE IKE compatibility, `length` controlled by `var.psk_length`).

The `health_check.target` attribute is a nested object ‚Äî must be set as `{ saved = <ip> }`, not a plain string. Always use the CPE inside IP (VTI address) as the health check target for all tunnels regardless of whether `customer_gw_ip` is set ‚Äî this is the correct approach for bidirectional health checks over the tunnel.

### Output in outputs.tf

Four `local_file` resources written on every `terraform apply`:
- `output/cpe-config.csv` ‚Äî sorted, deterministic CSV of all tunnel parameters
- `output/configure-tunnels.sh` ‚Äî ready-to-run ECOS configuration script for Mac/Linux (rendered from `aruba/configure-tunnels.tftpl`)
- `output/remove-tunnels.sh` ‚Äî rollback script for Mac/Linux (rendered from `aruba/remove-tunnels.tftpl`)
- `output/configure-tunnels.ps1` ‚Äî ready-to-run ECOS configuration script for Windows (rendered from `aruba/configure-tunnels.ps1.tftpl`)
- `output/remove-tunnels.ps1` ‚Äî rollback script for Windows (rendered from `aruba/remove-tunnels.ps1.tftpl`)

A `locals` block (`appliance_script_data`) groups `tunnel_definitions` by `ec_hostname` and pre-builds the full ECOS JSON payload per tunnel using `jsonencode()`. This is passed to `templatefile()`.

The `fqdn_id` (Local IKE Identifier) is constructed as:
```
${tunnel.id}.${var.cloudflare_conduit_id}.ipsec.cloudflare.com
```
The Cloudflare provider does **not** expose a computed `fqdn_id` attribute ‚Äî the format is `<tunnel-id>.<conduit-id>.ipsec.cloudflare.com` (not account ID).

The `tunnel_details` output includes these fields per tunnel:
`tunnel_name`, `tunnel_id`, `tunnel_label`, `site_name`, `cloudflare_endpoint`,
`customer_endpoint`, `interface_address`, `cf_inside_ip`, `cpe_inside_ip`,
`fqdn_id`, `ec_hostname`

## Key Conventions

- `customer_gw_ip` in `sites.csv` is informational only and no longer affects tunnel config ‚Äî with a mix of NAT'd and static sites, tracking which is which adds complexity for no benefit. The Cloudflare tunnel's `customer_endpoint` is always `null` (even known-static addresses can be dynamically NAT'd); tunnels are identified via the FQDN ID (`ike_id_local`), not a fixed source IP. The CPE-side `source` field in the ECOS payload is *always* embedded as `"0.0.0.0"` and resolved dynamically from Orchestrator at `configure-tunnels.sh`/`.ps1` run time for every site, regardless of `customer_gw_ip` ‚Äî `--orchestrator`/`--orch-token` (or `$ARUBA_API_TOKEN`) is required on every run, not just for NAT'd sites. The resolution prefers the interface's own `ipv4` over Orchestrator's `publicIp` field, since `source` must be an address actually bound to a local interface (see VTI/tunnel troubleshooting note below) ‚Äî NAT then transparently rewrites it to the public IP in flight.
- Health check target is always the CPE inside (VTI) IP ‚Äî not `customer_gw_ip`
- Tunnel names: `{site_name}-{pri|sec}`
- Supernet `/22` + 9 bits = /31 ‚Üí 512 subnets ‚Üí max 256 sites
- `site_index` in CSV must be unique, starting from 0; controls IP allocation independent of row order
- All health check and IPsec settings are per-account variables, not per-site
- `terraform.tfvars`, `output/`, `sites.csv.proposed`, and `*.tfstate*` are gitignored (contain secrets)

Scripts authenticate using an Orchestrator API token via the `X-Auth-Token` header.

- **All Orchestrator API operations** (site discovery, WAN IP resolution for NAT'd sites): `Site Admin, read-only` is sufficient. The Orchestrator is only queried for reads.
- **VTI creation** is done directly on the ECOS appliance (not the Orchestrator) ‚Äî no Orchestrator write access is needed.

Store the token in `.env` (gitignored) or set `ARUBA_API_TOKEN` in your environment:

### get_site_details.sh (primary sites.csv generator)

Queries Orchestrator for all appliances and writes `sites.csv.proposed`.

```bash
bash aruba/get_site_details.sh --orchestrator HOST [--ec-user USER] [--ec-pass PASS] [--sites HOST1,HOST2]
```

**What it collects per appliance:**
- `interfaceState` endpoint ‚Äî WAN interfaces (with public IP preference), LAN interfaces
- `subnets` endpoint ‚Äî advertised routes (`advert=true`), BGP-learned routes, OSPF-learned routes
- ECOS appliance API directly (if `--ec-user`/`--ec-pass` provided) ‚Äî BGP neighbor config/state

**`api_target` logic** (used as `ec_hostname` in CSV):
- EC-V appliances: `mgmt0` interface IP (dedicated management interface)
- Hardware appliances (EC-S, EC-10104): Orchestrator's management IP (`mgmt0` has no IP on hardware; management runs over LAN/WAN)

**`lan_subnets` source**: `/subnets` endpoint ‚Äî union of routes with `advert=true` (SD-WAN fabric-advertised) and routes with `local=true` on `lan*` interfaces (locally connected LAN subnets). The union covers both full SD-WAN fabric sites and simpler sites where LAN subnets are locally connected but not fabric-advertised. WAN/mgmt subnets are excluded because they won't have a `lan*` ifName.

### Orchestrator REST API endpoints used

| Method | Endpoint | Purpose |
|---|---|---|
| GET | `/gms/rest/appliance` | List all appliances (`id`=nePk, `hostName`, `IP`=mgmt IP) |
| GET | `/gms/rest/interfaceState?nePk={nePk}&cached=true` | Interface state (WAN/LAN IPs) |
| GET | `/gms/rest/subnets?nePk={nePk}` | Route table: advertised, BGP-learned, OSPF-learned routes |
| GET | `/gms/rest/vrrp?nePk={nePk}` | VRRP config (returns `[]` if unused) |
| GET | `/gms/rest/portForwarding?nePk={nePk}` | Port forwarding rules |
| GET | `/gms/rest/virtualif/vti?nePk={nePk}&cached=false` | Read VTI config (GET only ‚Äî writes return 405) |

**Not available via Orchestrator REST API**:
- BGP neighbor config/state (returns 404/405) ‚Äî query directly from each appliance via ECOS API with `--ec-user`/`--ec-pass`.
- VTI creation/modification ‚Äî `POST /gms/rest/virtualif/vti` returns **405 Method Not Allowed**. VTIs must be created directly on the ECOS appliance (see VTI creation below).

**`interfaceState` response structure** (important ‚Äî differs from old `get_appliances.sh` assumptions):
- Top-level keys: `sysConfig`, `scalars`, `ifInfo`, `addrstIp6`, `l3info`, `macIfs`, `availMACs`
- Interfaces are in `.ifInfo[]` array
- Interface fields: `ifname` (string, e.g., `"wan0"`, `"lan0"`, `"mgmt0"`), `oper` (bool), `ipv4` (string), `ipv4mask` (**integer** prefix length, not dotted-decimal), `publicIp` (string)
- `wan-if` and `lan-if` boolean fields exist but are **always false** on current firmware ‚Äî classify by `ifname` prefix instead

**`subnets` response structure**:
- `.subnets.entries[].state.prefix` ‚Äî CIDR (network address, not host IP)
- `.subnets.entries[].state.advert` ‚Äî true if advertised over SD-WAN fabric
- `.subnets.entries[].state.learned_bgp` / `.learned_ospf` ‚Äî dynamic routing source
- `.subnets.entries[].state.local` ‚Äî true if locally connected
- `.subnets.entries[].state.ifName` ‚Äî interface the route is associated with

### configure-tunnels.sh and remove-tunnels.sh (auto-generated by terraform apply)

Both scripts are rendered from templates in `aruba/` by `local_file` Terraform resources and regenerated on every `terraform apply` when tunnel IDs or the PSK change.

```bash
bash output/configure-tunnels.sh [--username USER] [--password PASS] [--orchestrator HOST] [--orch-token TOKEN] [--sites HOST1,HOST2] [--verify-ssl]
bash output/remove-tunnels.sh    [--username USER] [--password PASS] [--sites HOST1,HOST2] [--dry-run] [--verify-ssl]
```

**`configure-tunnels.sh`** (`aruba/configure-tunnels.tftpl`):
- One section per appliance: login ‚Üí POST both IPsec tunnels ‚Üí POST VTI on appliance ‚Üí logout
- PSK and Cloudflare tunnel IDs/IPs embedded at `terraform apply` time
- VTI creation uses the existing ECOS session ‚Äî no Orchestrator involvement
- `--orchestrator ORCH` is **required for every site** to resolve the appliance's current WAN IP at run time ‚Äî `customer_gw_ip` in `sites.csv` no longer exempts a site from this.
- `--orch-token TOKEN` (or `$ARUBA_API_TOKEN`) provides Orchestrator auth for WAN IP lookup
- Every tunnel's payload has `source=0.0.0.0` embedded; the script always patches it with the live WAN IP via `sed` before POSTing
- Treat as a secret (contains PSK); `output/` is gitignored

**`remove-tunnels.sh`** (`aruba/remove-tunnels.tftpl`):
- Rollback / uninstall counterpart to `configure-tunnels.sh`
- **Self-contained ‚Äî does NOT require terraform or terraform state** ‚Äî safe to run after `terraform destroy`
- GETs all passthrough tunnels from each appliance, matches by `alias` field, calls `deleteMultiple`
- After tunnel deletion, GETs all VTIs and DELETEs any with `.tunnel` matching our tunnel names
- Tunnel names are embedded at `terraform apply` time; no state lookup needed at removal time
- `--dry-run` shows what tunnels and VTIs would be removed without making any API calls
- `--sites` filters by appliance IP (comma-separated), consistent with `configure-tunnels.sh`

Both scripts use `--sites` by appliance IP/hostname (matching `ec_hostname` in `sites.csv`), NOT by site name.

### EdgeConnect REST API (ECOS)

Scripts connect **directly to each EdgeConnect appliance** (not via Orchestrator) using the ECOS REST API.

**Authentication:** Session cookie via `POST /login` with `{"user":"...","password":"..."}`.

**CRITICAL:** ECOS requires BOTH a session cookie AND a CSRF token on every subsequent call:
1. POST `/login` ‚Üí sets `vxoaSessionID` (HttpOnly) and `edgeosCsrfToken` (readable) cookies
2. Strip `#HttpOnly_` prefix from cookie jar: `sed -i.bak 's/^#HttpOnly_//' cookiejar`
3. Extract CSRF token: `awk '/edgeosCsrfToken/{print $NF}' cookiejar`
4. Send on all calls: `-b cookiejar -H "X-XSRF-TOKEN: <csrf_token>"`

Omitting `X-XSRF-TOKEN` returns `401 "You are not authenticated"` even after a successful login.

**Key endpoints used:**

| Method | Endpoint | Purpose |
|---|---|---|
| POST | `/rest/json/login` | Authenticate, receive session cookie |
| DELETE | `/rest/json/login` | Logout |
| GET | `/rest/json/thirdPartyTunnels/config` | List existing passthrough tunnels |
| POST | `/rest/json/thirdPartyTunnels/config` | Create/modify passthrough tunnels (key = tunnel alias) |
| POST | `/rest/json/thirdPartyTunnels/deleteMultiple` | Delete tunnels by ID list |
| GET | `/rest/json/virtualif/vti` | List VTIs (returns object keyed by `vtiN`) |
| POST | `/rest/json/virtualif/vti/{vtiN}` | Create VTI (key in URL path, body is VTI object) |
| DELETE | `/rest/json/virtualif/vti/{vtiN}` | Delete VTI |

**Passthrough tunnel key format:** The POST body key is the tunnel alias (e.g., `"hq-pri"`). Confirmed working on live appliance.

### EdgeConnect IPsec tunnel config (confirmed field names)

The POST body to `/thirdPartyTunnels/config` is a dict keyed by tunnel name:

```json
{
  "test-nat-site-pri": {
    "admin":             "up",
    "alias":             "test-nat-site-pri",
    "auto_mtu":          true,
    "gms_marked":        false,
    "ipsec_enable":      true,
    "ipsec_arc_window":  "disable",
    "presharedkey":      "<psk>",
    "mode":              "ipsec_ip",
    "nat_mode":          "none",
    "peername":          "Cloudflare_IPSec",
    "source":            "0.0.0.0",
    "destination":       "<cloudflare_anycast_ip>",
    "max_bw_auto":       true,
    "local_vrf":         0,
    "ipsec": {
      "ike_version":   2,
      "ike_ealg":      "aes256",
      "ike_aalg":      "sha256",
      "dhgroup":       "14",
      "pfs":           true,
      "pfsgroup":      "14",
      "ike_prf":       "auto",
      "ipsec_suite_b": "none",
      "id_type":       "ufqdn",
      "ike_id_local":  "<fqdn_id>",
      "ike_id_remote": "<cloudflare_anycast_ip>",
      "exchange_mode": "aggressive",
      "mode":          "tunnel",
      "esn":           false,
      "dpd_delay":     0,
      "dpd_retry":     3,
      "ike_lifetime":  0,
      "lifetime":      240,
      "lifebytes":     0,
      "security": {
        "ah":  { "algorithm": "sha256" },
        "esp": { "algorithm": "aes256" }
      }
    }
  }
}
```

**Key field notes (all confirmed from live appliance GET response):**
- `mode`: **`"ipsec_ip"`** ‚Äî correct passthrough IPsec mode. `"ipsec_udp"` and `"ipsec"` are overlay modes (HTTP 400). `"no_encap"` is non-IPsec only.
- `source`: Must be an address actually bound to the appliance's local WAN interface ‚Äî the appliance binds/originates IKE and ESP packets from this address, so it cannot be a NAT-translated public IP that isn't assigned to any local interface (that causes the tunnel to accept the config but never transmit ‚Äî zero bytes in both directions, `Oper: Down` forever). **Cannot be `""` (empty string) either ‚Äî causes HTTP 500.** The Terraform payload always embeds `"0.0.0.0"` as a placeholder regardless of `customer_gw_ip`; `configure-tunnels.sh`/`.ps1` always resolves the live WAN IP from Orchestrator at run time and patches it in before POSTing ‚Äî preferring the interface's own `ipv4` over Orchestrator's `publicIp` field, since upstream NAT transparently rewrites the local IP to the public one in flight. `--orchestrator` is therefore required on every run, for every site.
- `destination`: Cloudflare anycast IP (`cloudflare_endpoint` from terraform output)
- `nat_mode`: **`"none"`** ‚Äî `"auto"` is not a valid enum value and causes HTTP 500. NAT traversal is handled automatically by IKEv2 when needed.
- `ipsec_arc_window`: **`"disable"`** ‚Äî `"0"` is not a valid value and causes HTTP 500. Other valid values: `"8192"` (enabled). Disabled is correct for Cloudflare tunnels.
- `ipsec.dhgroup` / `ipsec.pfsgroup`: **string `"14"`** (not integer `14`) ‚Äî type mismatch causes HTTP 500. Group 14 = 2048-bit MODP.
- `ipsec.security.ah.algorithm`: **`"sha256"`** ‚Äî SHA-256 authentication via AH. Confirmed from live appliance.
- `ipsec.security.esp.algorithm`: **`"aes256"`** (AES-CBC-256) ‚Äî encryption only; auth is carried in AH. `"aes256-sha256"` combined format is invalid (HTTP 500).
- `ipsec.mode`: **`"tunnel"`** ‚Äî standard for site-to-site IPsec.
- `ipsec.exchange_mode`: **`"aggressive"`** (not `"main"`) ‚Äî confirmed from live appliance.
- `ipsec.id_type`: **`"ufqdn"`** (user FQDN, not `"fqdn"`) ‚Äî confirmed from live appliance.
- `ipsec.ike_prf`: **`"auto"`** ‚Äî confirmed from live appliance.
- `ipsec.ipsec_suite_b`: **`"none"`** ‚Äî confirmed from live appliance.
- `peername`: **`"Cloudflare_IPSec"`** ‚Äî the Peer/Service label shown in the ECOS UI.
- `ipsec.lifetime`: **`240`** ‚Äî IPsec SA lifetime in seconds. Confirmed from live appliance.
- `ipsec.ike_id_local`: the `fqdn_id`: `<tunnel-id>.<conduit-id>.ipsec.cloudflare.com`
- `ipsec.ike_id_remote`: the Cloudflare anycast IP (`cloudflare_endpoint`) ‚Äî identifies the remote peer

### VTI creation (ECOS appliance, required after tunnel creation)

After each passthrough tunnel is created, a VTI (Virtual Tunnel Interface) must be created **directly on the ECOS appliance** ‚Äî not via Orchestrator. The Orchestrator VTI endpoint (`/gms/rest/virtualif/vti`) is **read-only** ‚Äî all write methods return **405 Method Not Allowed**.

VTI creation uses the existing ECOS session (same cookie + CSRF token from login) ‚Äî no additional auth needed.

**GET:** `GET /rest/json/virtualif/vti` ‚Äî returns object keyed by `vtiN`
**CREATE:** `POST /rest/json/virtualif/vti/{vtiN}` ‚Äî key in URL path, body is VTI object
**DELETE:** `DELETE /rest/json/virtualif/vti/{vtiN}`

**VTI key format:** `vti` + integer only (e.g., `vti110`, `vti111`) ‚Äî alphanumeric keys are rejected with HTTP 500.

**Auto-increment:** GET existing VTIs, find max `vtiN` number, use max+1. Start at 110 if none exist.

**Confirmed working payload (minimal ‚Äî ECOS-UI-visible fields only):**
```json
{
  "admin": true,
  "ipaddr": "<cpe_inside_ip>",
  "nmask": <prefix_len_int>,
  "side": "wan",
  "tunnel": "<tunnel_name>"
}
```

**Critical field notes:**
- Extra fields (`auto_distribute`, `behindNAT`, `gms_marked`, `ipaddr_alias`, `label`, `label_alias`, `nmask_alias`, `role_id`, `vrf_id`, `zone`) cause HTTP 500 on this firmware ‚Äî stick to the 5 minimal fields above.
- `side`: must be `"wan"`, not `"lan"` ‚Äî a LAN-side VTI accepts the POST (HTTP 200) but never binds to the tunnel interface, so the tunnel configures without error but never comes up.
- `nmask`: prefix length as integer (e.g., `31`), not dotted-decimal.
- `tunnel`: must match the passthrough tunnel alias exactly.
- HTTP 200 + `{}` body on success.
- Both `configure-tunnels.tftpl` (bash) and `configure-tunnels.ps1.tftpl` (PowerShell) must stay in sync on this payload ‚Äî a prior fix (trimming to minimal fields) was applied only to the PowerShell template and missed on bash, causing the two platforms to diverge.

**Idempotency:** GET VTIs first, check if any entry has `.tunnel == <tunnel_name>`. Skip creation if found.

**Order matters:** Create passthrough tunnel on appliance FIRST, then create VTI on same appliance.

### Workflow for new deployments

1. **Generate sites.csv from Orchestrator:**
   ```bash
   bash aruba/get_site_details.sh --orchestrator HOST
   # Review sites.csv.proposed:
   #   - customer_gw_ip is informational only ‚Äî no need to fill it in or worry about
   #     NAT'd vs static; source is always resolved dynamically at configure-tunnels time
   #   - STL-ARMS style dual-WAN: script picks first publicIp; edit if wrong WAN should be used
   #   - ec_hostname = api_target IP; replace with DNS name if preferred
   #   - site_index is auto-assigned alphabetically ‚Äî do not change once applied
   cp sites.csv.proposed sites.csv
   ```

2. **Create Cloudflare tunnels and routes:**
   ```bash
   terraform plan
   terraform apply -parallelism=1
   # This also generates output/configure-tunnels.sh
   ```

3. **Configure EdgeConnect appliances and create VTIs:**
   ```bash
   # Orchestrator is required for every site -- WAN IP is always resolved at run time:
   bash output/configure-tunnels.sh --username admin \
     --orchestrator ORCH_HOST --orch-token TOKEN
   # Or with $ARUBA_API_TOKEN set:
   bash output/configure-tunnels.sh --username admin --orchestrator ORCH_HOST

   # Filter to specific appliances (by ec_hostname IP):
   bash output/configure-tunnels.sh --username admin --orchestrator ORCH_HOST --sites 10.20.0.2,10.20.0.3
   ```
   VTIs are created directly on each appliance via ECOS API (same session as tunnel creation).

4. **Rollback EdgeConnect appliances (if needed):**
   ```bash
   bash output/remove-tunnels.sh --dry-run              # preview
   bash output/remove-tunnels.sh --username admin       # execute
   # remove-tunnels.sh is self-contained ‚Äî works before or after terraform destroy
   ```

5. **Destroy Cloudflare resources:**
   ```bash
   terraform destroy -parallelism=1
   ```

### Cloudflare API rate limiting

The Magic WAN API uses a **per-account write lock** ‚Äî concurrent requests from Terraform's default parallel execution cause `429` errors with code `1059` ("account is being modified elsewhere"). Always use:
```bash
terraform apply -parallelism=1
terraform destroy -parallelism=1
```
Transient `400` errors (code `1012`, database connection reset) are Cloudflare-side infrastructure glitches ‚Äî just re-run `terraform apply -parallelism=1` to retry failed resources.

When replacing all sites (new `sites.csv` with different `site_index` values), always `terraform destroy` first. Terraform creates new resources before destroying old ones, and Cloudflare rejects duplicate `/31` inside addresses.

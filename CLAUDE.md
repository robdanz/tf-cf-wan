# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Terraform project managing Cloudflare Magic WAN IPsec tunnels at scale. Creates 2 IPsec tunnels per site (primary + secondary, one to each Cloudflare Anycast IP), creates static routes for each site's LAN subnets, and generates `output/configure-tunnels.sh` to configure the corresponding IPsec tunnels on Aruba EdgeConnect appliances via ECOS REST API.

## Commands

```bash
# Terraform (Cloudflare side)
terraform init                    # Download providers (cloudflare v5, random, local)
terraform validate                # Check syntax
terraform plan
terraform apply -parallelism=1    # REQUIRED: Cloudflare Magic WAN API uses a per-account write lock;
                                  # parallel requests cause 429 "account is being modified elsewhere"
terraform output -raw tunnel_psk                    # Retrieve the shared PSK
terraform output -json tunnel_details               # All tunnel data as JSON
terraform state show 'cloudflare_magic_wan_ipsec_tunnel.tunnels["<site>-pri"]'

# Aruba EdgeConnect — site discovery
bash aruba/get_site_details.sh --orchestrator HOST  # Query Orchestrator → sites.csv.proposed
bash aruba/demo_orchestrator.sh [--output FILE]     # Simulate Orchestrator for demo (10 NAT'd sites)

# Aruba EdgeConnect — configure/remove (Mac/Linux, generated by terraform apply)
bash output/configure-tunnels.sh --username admin                          # static sites
bash output/configure-tunnels.sh --username admin --orchestrator ORCH_HOST # NAT'd sites
bash output/remove-tunnels.sh --username admin

# Aruba EdgeConnect — configure/remove (Windows PowerShell)
.\aruba\configure_tunnels.ps1 [-DryRun] [-Sites "site1,site2"]
.\aruba\remove_tunnels.ps1    [-DryRun] [-Sites "site1,site2"]
```

## Architecture

**Full data flow:**
```
aruba/get_site_details.sh
    │
    └─► sites.csv.proposed  (review + cp to sites.csv)
            │
            ├─► terraform apply -parallelism=1
            │       │
            │       ├─► Cloudflare: IPsec tunnels (2 per site)
            │       ├─► Cloudflare: static routes (1 per site subnet per tunnel)
            │       ├─► output/cpe-config.csv
            │       └─► output/configure-tunnels.sh  ◄── auto-generated ECOS script
            │
            └─► bash output/configure-tunnels.sh --username admin
                    └─► EdgeConnect appliances (IPsec tunnels configured via ECOS API)
```

**`output/configure-tunnels.sh` is generated by `terraform apply`** from `aruba/configure-tunnels.tftpl` — no manual script generation step needed. It has the PSK and tunnel IDs embedded.

### Core transformation in locals.tf

1. `csvdecode()` reads `sites.csv` into a site map keyed by `site_name`
2. Each site is flattened into 2 tunnel entries (`<site>-pri`, `<site>-sec`) via `flatten()`
3. `/31` inside addresses are allocated from the supernet using `cidrsubnet(supernet, 9, site_index * 2 + offset)` — the explicit `site_index` column (not row position) controls IP allocation so CSV reordering is safe
4. Within each /31: Cloudflare = `cidrhost(..., 0)` (even/lower), CPE = `cidrhost(..., 1)` (odd/upper)
5. Automatic return routing is enabled on all tunnels (`automatic_return_routing = true`) — no static routes needed

### Resource pattern in tunnels.tf

Single `cloudflare_magic_wan_ipsec_tunnel` resource with `for_each = local.tunnel_definitions`. One `random_password` generates a shared PSK (`special = false` for CPE IKE compatibility, `length` controlled by `var.psk_length`).

The `health_check.target` attribute is a nested object — must be set as `{ saved = <ip> }`, not a plain string. Always use the CPE inside IP (VTI address) as the health check target for all tunnels regardless of whether `customer_gw_ip` is set — this is the correct approach for bidirectional health checks over the tunnel.

### Output in outputs.tf

Two `local_file` resources written on every `terraform apply`:
- `output/cpe-config.csv` — sorted, deterministic CSV of all tunnel parameters
- `output/configure-tunnels.sh` — ready-to-run ECOS configuration script (rendered from `aruba/configure-tunnels.tftpl`)

A `locals` block (`appliance_script_data`) groups `tunnel_definitions` by `ec_hostname` and pre-builds the full ECOS JSON payload per tunnel using `jsonencode()`. This is passed to `templatefile()`.

The `fqdn_id` (Local IKE Identifier) is constructed as:
```
${tunnel.id}.${var.cloudflare_conduit_id}.ipsec.cloudflare.com
```
The Cloudflare provider does **not** expose a computed `fqdn_id` attribute — the format is `<tunnel-id>.<conduit-id>.ipsec.cloudflare.com` (not account ID).

The `tunnel_details` output includes these fields per tunnel:
`tunnel_name`, `tunnel_id`, `tunnel_label`, `site_name`, `cloudflare_endpoint`,
`customer_endpoint`, `interface_address`, `cf_inside_ip`, `cpe_inside_ip`,
`fqdn_id`, `ec_hostname`

## sites.csv Columns

| Column | Required | Description |
|---|---|---|
| `site_name` | yes | Unique site identifier, used as tunnel name prefix |
| `site_index` | yes | Unique integer (0-based); controls /31 IP allocation — never reuse or reorder |
| `customer_gw_ip` | no | CPE WAN IP; blank for NAT'd/dynamic sites |
| `ec_hostname` | yes | EdgeConnect API target IP/hostname — used by `configure-tunnels.sh` |

`ec_hostname` should be populated from `api_target` in `get_site_details.sh` output:
- EC-V appliances: `mgmt0` interface IP
- Hardware appliances (EC-S, EC-10104): Orchestrator management IP (mgmt0 is unconfigured on hardware)

## Variables and tfvars

All account-specific and tuneable values live in `terraform.tfvars` (gitignored). No account-specific defaults are hardcoded in `variables.tf`.

| Variable | Description |
|---|---|
| `cloudflare_api_token` | API token (Magic WAN: Edit + Account Settings: Read) |
| `cloudflare_account_id` | Cloudflare account ID |
| `cloudflare_conduit_id` | Conduit ID used in IKE FQDN identifier |
| `anycast_ip_1` | Primary Cloudflare Anycast IP (from account team) |
| `anycast_ip_2` | Secondary Cloudflare Anycast IP (from account team) |
| `tunnel_supernet` | Supernet for /31 inside address allocation |
| `health_check_enabled` | `true` / `false` |
| `health_check_direction` | `unidirectional` / `bidirectional` — default `bidirectional` |
| `health_check_type` | `reply` / `request` — default `request` |
| `health_check_rate` | `low` / `mid` / `high` — default `mid` |
| `replay_protection` | `true` / `false` — disable if CPE has IKE compatibility issues |

**Note:** `terraform.tfvars` overrides these defaults. If health check changes don't take effect, check for stale values there (e.g. `health_check_direction = "unidirectional"`). The correct production values are `bidirectional` / `request` / `mid`.

## Key Conventions

- `customer_gw_ip` is optional — blank for NAT'd/dynamic CPE; sets `customer_endpoint = null`
- Health check target is always the CPE inside (VTI) IP — not `customer_gw_ip`
- Tunnel names: `{site_name}-{pri|sec}`
- Supernet `/22` + 9 bits = /31 → 512 subnets → max 256 sites
- `site_index` in CSV must be unique, starting from 0; controls IP allocation independent of row order
- All health check and IPsec settings are per-account variables, not per-site
- `terraform.tfvars`, `output/`, `sites.csv.proposed`, and `*.tfstate*` are gitignored (contain secrets)

## Cloudflare API Token Scopes

Account-level: **Magic WAN: Edit** + **Account Settings: Read**

## Aruba Orchestrator API Token

Scripts authenticate using an Orchestrator API token via the `X-Auth-Token` header.

- **All Orchestrator API operations** (site discovery, WAN IP resolution for NAT'd sites): `Site Admin, read-only` is sufficient. The Orchestrator is only queried for reads.
- **VTI creation** is done directly on the ECOS appliance (not the Orchestrator) — no Orchestrator write access is needed.

Store the token in `.env` (gitignored) or set `ARUBA_API_TOKEN` in your environment:
```bash
export ARUBA_API_TOKEN="your-token-here"
```

### get_site_details.sh (primary sites.csv generator)

Queries Orchestrator for all appliances and writes `sites.csv.proposed`.

```bash
bash aruba/get_site_details.sh --orchestrator HOST [--ec-user USER] [--ec-pass PASS] [--sites HOST1,HOST2]
```

**What it collects per appliance:**
- `interfaceState` endpoint — WAN interfaces (with public IP preference), LAN interfaces
- `subnets` endpoint — advertised routes (`advert=true`), BGP-learned routes, OSPF-learned routes
- ECOS appliance API directly (if `--ec-user`/`--ec-pass` provided) — BGP neighbor config/state

**`api_target` logic** (used as `ec_hostname` in CSV):
- EC-V appliances: `mgmt0` interface IP (dedicated management interface)
- Hardware appliances (EC-S, EC-10104): Orchestrator's management IP (`mgmt0` has no IP on hardware; management runs over LAN/WAN)

**`lan_subnets` source**: `/subnets` endpoint — union of routes with `advert=true` (SD-WAN fabric-advertised) and routes with `local=true` on `lan*` interfaces (locally connected LAN subnets). The union covers both full SD-WAN fabric sites and simpler sites where LAN subnets are locally connected but not fabric-advertised. WAN/mgmt subnets are excluded because they won't have a `lan*` ifName.

### Orchestrator REST API endpoints used

| Method | Endpoint | Purpose |
|---|---|---|
| GET | `/gms/rest/appliance` | List all appliances (`id`=nePk, `hostName`, `IP`=mgmt IP) |
| GET | `/gms/rest/interfaceState?nePk={nePk}&cached=true` | Interface state (WAN/LAN IPs) |
| GET | `/gms/rest/subnets?nePk={nePk}` | Route table: advertised, BGP-learned, OSPF-learned routes |
| GET | `/gms/rest/vrrp?nePk={nePk}` | VRRP config (returns `[]` if unused) |
| GET | `/gms/rest/portForwarding?nePk={nePk}` | Port forwarding rules |
| GET | `/gms/rest/virtualif/vti?nePk={nePk}&cached=false` | Read VTI config (GET only — writes return 405) |

**Not available via Orchestrator REST API**:
- BGP neighbor config/state (returns 404/405) — query directly from each appliance via ECOS API with `--ec-user`/`--ec-pass`.
- VTI creation/modification — `POST /gms/rest/virtualif/vti` returns **405 Method Not Allowed**. VTIs must be created directly on the ECOS appliance (see VTI creation below).

**`interfaceState` response structure** (important — differs from old `get_appliances.sh` assumptions):
- Top-level keys: `sysConfig`, `scalars`, `ifInfo`, `addrstIp6`, `l3info`, `macIfs`, `availMACs`
- Interfaces are in `.ifInfo[]` array
- Interface fields: `ifname` (string, e.g., `"wan0"`, `"lan0"`, `"mgmt0"`), `oper` (bool), `ipv4` (string), `ipv4mask` (**integer** prefix length, not dotted-decimal), `publicIp` (string)
- `wan-if` and `lan-if` boolean fields exist but are **always false** on current firmware — classify by `ifname` prefix instead

**`subnets` response structure**:
- `.subnets.entries[].state.prefix` — CIDR (network address, not host IP)
- `.subnets.entries[].state.advert` — true if advertised over SD-WAN fabric
- `.subnets.entries[].state.learned_bgp` / `.learned_ospf` — dynamic routing source
- `.subnets.entries[].state.local` — true if locally connected
- `.subnets.entries[].state.ifName` — interface the route is associated with

### configure-tunnels.sh and remove-tunnels.sh (auto-generated by terraform apply)

Both scripts are rendered from templates in `aruba/` by `local_file` Terraform resources and regenerated on every `terraform apply` when tunnel IDs or the PSK change.

```bash
bash output/configure-tunnels.sh [--username USER] [--password PASS] [--orchestrator HOST] [--orch-token TOKEN] [--sites HOST1,HOST2] [--verify-ssl]
bash output/remove-tunnels.sh    [--username USER] [--password PASS] [--sites HOST1,HOST2] [--dry-run] [--verify-ssl]
```

**`configure-tunnels.sh`** (`aruba/configure-tunnels.tftpl`):
- One section per appliance: login → POST both IPsec tunnels → POST VTI on appliance → logout
- PSK and Cloudflare tunnel IDs/IPs embedded at `terraform apply` time
- VTI creation uses the existing ECOS session — no Orchestrator involvement
- `--orchestrator ORCH` is **only needed for NAT'd/dynamic sites** (blank `customer_gw_ip`) to resolve the appliance's current WAN IP at run time. Not needed for static sites.
- `--orch-token TOKEN` (or `$ARUBA_API_TOKEN`) provides Orchestrator auth for WAN IP lookup
- For NAT'd sites, the payload has `source=0.0.0.0` embedded; the script patches it with the live WAN IP via `sed` before POSTing
- Treat as a secret (contains PSK); `output/` is gitignored

**`remove-tunnels.sh`** (`aruba/remove-tunnels.tftpl`):
- Rollback / uninstall counterpart to `configure-tunnels.sh`
- **Self-contained — does NOT require terraform or terraform state** — safe to run after `terraform destroy`
- GETs all passthrough tunnels from each appliance, matches by `alias` field, calls `deleteMultiple`
- After tunnel deletion, GETs all VTIs and DELETEs any with `.tunnel` matching our tunnel names
- Tunnel names are embedded at `terraform apply` time; no state lookup needed at removal time
- `--dry-run` shows what tunnels and VTIs would be removed without making any API calls
- `--sites` filters by appliance IP (comma-separated), consistent with `configure-tunnels.sh`

Both scripts use `--sites` by appliance IP/hostname (matching `ec_hostname` in `sites.csv`), NOT by site name.

### EdgeConnect REST API (ECOS)

Scripts connect **directly to each EdgeConnect appliance** (not via Orchestrator) using the ECOS REST API.

**Authentication:** Session cookie via `POST /login` with `{"user":"...","password":"..."}`.

**CRITICAL:** ECOS requires BOTH a session cookie AND a CSRF token on every subsequent call:
1. POST `/login` → sets `vxoaSessionID` (HttpOnly) and `edgeosCsrfToken` (readable) cookies
2. Strip `#HttpOnly_` prefix from cookie jar: `sed -i.bak 's/^#HttpOnly_//' cookiejar`
3. Extract CSRF token: `awk '/edgeosCsrfToken/{print $NF}' cookiejar`
4. Send on all calls: `-b cookiejar -H "X-XSRF-TOKEN: <csrf_token>"`

Omitting `X-XSRF-TOKEN` returns `401 "You are not authenticated"` even after a successful login.

**Key endpoints used:**

| Method | Endpoint | Purpose |
|---|---|---|
| POST | `/rest/json/login` | Authenticate, receive session cookie |
| DELETE | `/rest/json/login` | Logout |
| GET | `/rest/json/thirdPartyTunnels/config` | List existing passthrough tunnels |
| POST | `/rest/json/thirdPartyTunnels/config` | Create/modify passthrough tunnels (key = tunnel alias) |
| POST | `/rest/json/thirdPartyTunnels/deleteMultiple` | Delete tunnels by ID list |
| GET | `/rest/json/virtualif/vti` | List VTIs (returns object keyed by `vtiN`) |
| POST | `/rest/json/virtualif/vti/{vtiN}` | Create VTI (key in URL path, body is VTI object) |
| DELETE | `/rest/json/virtualif/vti/{vtiN}` | Delete VTI |

**Passthrough tunnel key format:** The POST body key is the tunnel alias (e.g., `"hq-pri"`). Confirmed working on live appliance.

### EdgeConnect IPsec tunnel config (confirmed field names)

The POST body to `/thirdPartyTunnels/config` is a dict keyed by tunnel name:

```json
{
  "test-nat-site-pri": {
    "admin":             "up",
    "alias":             "test-nat-site-pri",
    "auto_mtu":          true,
    "gms_marked":        false,
    "ipsec_enable":      true,
    "ipsec_arc_window":  "disable",
    "presharedkey":      "<psk>",
    "mode":              "ipsec_ip",
    "nat_mode":          "none",
    "peername":          "Cloudflare_IPSec",
    "source":            "0.0.0.0",
    "destination":       "<cloudflare_anycast_ip>",
    "max_bw_auto":       true,
    "local_vrf":         0,
    "ipsec": {
      "ike_version":   2,
      "ike_ealg":      "aes256",
      "ike_aalg":      "sha256",
      "dhgroup":       "14",
      "pfs":           true,
      "pfsgroup":      "14",
      "ike_prf":       "auto",
      "ipsec_suite_b": "none",
      "id_type":       "ufqdn",
      "ike_id_local":  "<fqdn_id>",
      "ike_id_remote": "<cloudflare_anycast_ip>",
      "exchange_mode": "aggressive",
      "mode":          "tunnel",
      "esn":           false,
      "dpd_delay":     0,
      "dpd_retry":     3,
      "ike_lifetime":  0,
      "lifetime":      240,
      "lifebytes":     0,
      "security": {
        "ah":  { "algorithm": "sha256" },
        "esp": { "algorithm": "aes256" }
      }
    }
  }
}
```

**Key field notes (all confirmed from live appliance GET response):**
- `mode`: **`"ipsec_ip"`** — correct passthrough IPsec mode. `"ipsec_udp"` and `"ipsec"` are overlay modes (HTTP 400). `"no_encap"` is non-IPsec only.
- `source`: Appliance local WAN interface IP. **Cannot be `""` (empty string) — causes HTTP 500.** For NAT'd sites (blank `customer_gw_ip`), the Terraform payload embeds `"0.0.0.0"` as a placeholder; `configure-tunnels.sh` resolves the live WAN IP from Orchestrator at run time and patches it via `sed` before POSTing. Requires `--orchestrator` when `customer_gw_ip` is blank.
- `destination`: Cloudflare anycast IP (`cloudflare_endpoint` from terraform output)
- `nat_mode`: **`"none"`** — `"auto"` is not a valid enum value and causes HTTP 500. NAT traversal is handled automatically by IKEv2 when needed.
- `ipsec_arc_window`: **`"disable"`** — `"0"` is not a valid value and causes HTTP 500. Other valid values: `"8192"` (enabled). Disabled is correct for Cloudflare tunnels.
- `ipsec.dhgroup` / `ipsec.pfsgroup`: **string `"14"`** (not integer `14`) — type mismatch causes HTTP 500. Group 14 = 2048-bit MODP.
- `ipsec.security.ah.algorithm`: **`"sha256"`** — SHA-256 authentication via AH. Confirmed from live appliance.
- `ipsec.security.esp.algorithm`: **`"aes256"`** (AES-CBC-256) — encryption only; auth is carried in AH. `"aes256-sha256"` combined format is invalid (HTTP 500).
- `ipsec.mode`: **`"tunnel"`** — standard for site-to-site IPsec.
- `ipsec.exchange_mode`: **`"aggressive"`** (not `"main"`) — confirmed from live appliance.
- `ipsec.id_type`: **`"ufqdn"`** (user FQDN, not `"fqdn"`) — confirmed from live appliance.
- `ipsec.ike_prf`: **`"auto"`** — confirmed from live appliance.
- `ipsec.ipsec_suite_b`: **`"none"`** — confirmed from live appliance.
- `peername`: **`"Cloudflare_IPSec"`** — the Peer/Service label shown in the ECOS UI.
- `ipsec.lifetime`: **`240`** — IPsec SA lifetime in seconds. Confirmed from live appliance.
- `ipsec.ike_id_local`: the `fqdn_id`: `<tunnel-id>.<conduit-id>.ipsec.cloudflare.com`
- `ipsec.ike_id_remote`: the Cloudflare anycast IP (`cloudflare_endpoint`) — identifies the remote peer

### VTI creation (ECOS appliance, required after tunnel creation)

After each passthrough tunnel is created, a VTI (Virtual Tunnel Interface) must be created **directly on the ECOS appliance** — not via Orchestrator. The Orchestrator VTI endpoint (`/gms/rest/virtualif/vti`) is **read-only** — all write methods return **405 Method Not Allowed**.

VTI creation uses the existing ECOS session (same cookie + CSRF token from login) — no additional auth needed.

**GET:** `GET /rest/json/virtualif/vti` — returns object keyed by `vtiN`
**CREATE:** `POST /rest/json/virtualif/vti/{vtiN}` — key in URL path, body is VTI object
**DELETE:** `DELETE /rest/json/virtualif/vti/{vtiN}`

**VTI key format:** `vti` + integer only (e.g., `vti110`, `vti111`) — alphanumeric keys are rejected with HTTP 500.

**Auto-increment:** GET existing VTIs, find max `vtiN` number, use max+1. Start at 110 if none exist.

**Confirmed working payload:**
```json
{
  "admin": true,
  "auto_distribute": true,
  "behindNAT": "none",
  "gms_marked": false,
  "ipaddr": "<cpe_inside_ip>",
  "ipaddr_alias": "0.0.0.0",
  "label": "",
  "label_alias": "",
  "nmask": <prefix_len_int>,
  "nmask_alias": 0,
  "role_id": 0,
  "side": "lan",
  "tunnel": "<tunnel_name>",
  "vrf_id": 0,
  "zone": 0
}
```

**Critical field notes:**
- `behindNAT`: all caps — `behindNat` (camelCase) causes HTTP 500.
- `nmask`: prefix length as integer (e.g., `31`), not dotted-decimal.
- `tunnel`: must match the passthrough tunnel alias exactly.
- `label`: can be empty string `""` — the numeric nePk label is not required.
- HTTP 200 + `{}` body on success.

**Idempotency:** GET VTIs first, check if any entry has `.tunnel == <tunnel_name>`. Skip creation if found.

**Order matters:** Create passthrough tunnel on appliance FIRST, then create VTI on same appliance.

### Workflow for new deployments

1. **Generate sites.csv from Orchestrator:**
   ```bash
   bash aruba/get_site_details.sh --orchestrator HOST
   # Review sites.csv.proposed:
   #   - Verify customer_gw_ip — blank = NAT/dynamic (intentional or needs filling)
   #   - STL-ARMS style dual-WAN: script picks first publicIp; edit if wrong WAN should be used
   #   - ec_hostname = api_target IP; replace with DNS name if preferred
   #   - site_index is auto-assigned alphabetically — do not change once applied
   cp sites.csv.proposed sites.csv
   ```

2. **Create Cloudflare tunnels and routes:**
   ```bash
   terraform plan
   terraform apply -parallelism=1
   # This also generates output/configure-tunnels.sh
   ```

3. **Configure EdgeConnect appliances and create VTIs:**
   ```bash
   # Static sites (customer_gw_ip populated) — no Orchestrator needed:
   bash output/configure-tunnels.sh --username admin

   # NAT'd/dynamic sites (blank customer_gw_ip) — Orchestrator required for WAN IP lookup:
   bash output/configure-tunnels.sh --username admin \
     --orchestrator ORCH_HOST --orch-token TOKEN
   # Or with $ARUBA_API_TOKEN set:
   bash output/configure-tunnels.sh --username admin --orchestrator ORCH_HOST

   # Filter to specific appliances (by ec_hostname IP):
   bash output/configure-tunnels.sh --username admin --sites 10.20.0.2,10.20.0.3
   ```
   VTIs are created directly on each appliance via ECOS API (same session as tunnel creation).

4. **Rollback EdgeConnect appliances (if needed):**
   ```bash
   bash output/remove-tunnels.sh --dry-run              # preview
   bash output/remove-tunnels.sh --username admin       # execute
   # remove-tunnels.sh is self-contained — works before or after terraform destroy
   ```

5. **Destroy Cloudflare resources:**
   ```bash
   terraform destroy -parallelism=1
   ```

### Cloudflare API rate limiting

The Magic WAN API uses a **per-account write lock** — concurrent requests from Terraform's default parallel execution cause `429` errors with code `1059` ("account is being modified elsewhere"). Always use:
```bash
terraform apply -parallelism=1
terraform destroy -parallelism=1
```
Transient `400` errors (code `1012`, database connection reset) are Cloudflare-side infrastructure glitches — just re-run `terraform apply -parallelism=1` to retry failed resources.

When replacing all sites (new `sites.csv` with different `site_index` values), always `terraform destroy` first. Terraform creates new resources before destroying old ones, and Cloudflare rejects duplicate `/31` inside addresses.
